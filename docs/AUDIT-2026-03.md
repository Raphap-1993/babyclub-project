## Hallazgos principales
### H1. El gate global de tipos no existe realmente
- Severidad: `Alta`
- Evidencia:
  - `package.json` define `"check-types": "turbo run check-types"`
  - `turbo.json` no define ninguna task `check-types`
- Impacto comercial:
  - baja confianza en PR gate / CI
  - descuento inmediato en due diligence

### H2. El typecheck del backoffice falla por configuracion de tests
- Severidad: `Alta`
- Evidencia:
  - `apps/backoffice/tsconfig.json` incluye `**/*.ts` y `**/*.tsx`
  - eso arrastra `app/admin/events/__tests__/QRStatsTable.test.tsx`
  - faltan `@testing-library/react` y tipos globales de runner
- Impacto comercial:
  - el buyer interpreta que el proyecto depende del autor para compilar limpio
  - reduce capacidad de operacion por tercero

### H3. Hay una regresion visible en el flujo de mesas
- Severidad: `Alta`
- Evidencia:
  - `pnpm test` falla en `apps/landing/app/api/tables/route.test.ts`
  - la ruta devuelve `404` cuando `eventStateResult.data` no existe en `apps/landing/app/api/tables/route.ts`
- Impacto comercial:
  - mesas/box con combo obligatorio es core del negocio
  - no conviene ofrecer venta premium mientras ese flujo no vuelva a verde

### H4. La arquitectura objetivo V2 esta definida, pero no implementada
- Severidad: `Media`
- Evidencia:
  - `docs/ARCHITECTURE_V2.md` propone `apps/api-v2`, `packages/domain` y `packages/db`
  - esos paquetes/apps aun no existen en el repo
- Impacto comercial:
  - el proyecto es vendible como V1 establecida
  - todavia no justifica discurso de plataforma modular “lista para escalar” sin descuento

### H5. Observabilidad y rate limiting todavia no son de nivel multi-instancia
- Severidad: `Media`
- Evidencia:
  - `docs/SECURITY.md` declara rate limiting `in-memory`
  - `apps/backoffice/app/api/logs/logger.ts` escribe en `process_logs`, pero no adjunta `correlation_id`
- Impacto comercial:
  - aceptable para fase actual
  - insuficiente para prometer alta disponibilidad o auditoria premium enterprise

### H6. No hay workflow CI visible en GitHub
- Severidad: `Media`
- Evidencia:
  - no se encontraron archivos bajo `.github/workflows`
- Impacto comercial:
  - obliga a demostrar calidad manualmente
  - vuelve la operacion mas dependiente del fundador

- Traduccion practica:
  - un rebuild por un equipo serio no luce como “2 semanas”
  - aun con equipo nearshore, el costo real de reponer este activo probablemente es varias decenas de miles de dolares

### 2. Ajuste por riesgo tecnico
- Restar valor por:

  - `2` tests fallando en flujo de mesas
  - ausencia de CI visible

## Como alquilarlo mejor que venderlo hoy
## Tesis
- Comercialmente, hoy parece mejor `alquilar/licenciar` primero y `vender` despues.
- Razón:
  - conviertes codigo en ingreso recurrente
  - demuestras demanda
  - subes tu multiple
  - reduces la discusion subjetiva sobre “cuanto vale el software”

## Modelos de renta/licencia recomendados
### Modelo 1. White-label con setup + mensualidad
- Ideal para organizadores medianos o venues
- Precio sugerido:
  - setup: `USD 2k - 8k`
  - mensual: `USD 300 - 1,500`
  - soporte premium opcional: `USD 200 - 800` extra
- Cuando sirve:
  - marca propia
  - un organizador / venue
  - poco volumen inicial

### Modelo 2. Setup + fee por evento + fee transaccional
- Ideal para nightlife y productores con estacionalidad
- Precio sugerido:
  - onboarding: `USD 1k - 5k`
  - fee por evento: `USD 200 - 1,000`
  - fee variable: `0.5% - 2.5%` GMV o monto fijo por ticket
- Ventaja:
  - baja barrera de entrada
  - capturas upside cuando el evento funciona

### Modelo 3. Licencia anual enterprise
- Ideal para grupo de venues, ticketera local o agencia
- Precio sugerido:
  - licencia anual: `USD 12k - 30k`
  - onboarding / customizacion: aparte
  - SLA, soporte y nuevas features: aparte
- Cuando sirve:
  - multi-organizador
  - operacion continua
  - buyer quiere capex/opex predecible
